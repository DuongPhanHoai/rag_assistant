Student Management RAG with SQLite, AutoSchemaKG, and Neo4j

Recommended Dataset

Use Student Management because it naturally mixes structured analytics, policy documents, advising notes, and relationship-heavy graph questions.

  • Structured SQLite tables: students, courses, enrollments, attendance, assessments, fees, advising_cases.
  • Unstructured documents for AutoSchemaKG extraction: advising notes, course descriptions, scholarship policy, attendance policy, intervention playbooks.
  • Example questions:
    • Which students are at risk this term and why?
    • Show average grade by course and explain weak areas.
    • Who qualifies for scholarship support based on GPA, attendance, and fee status?
    • Create a chart of attendance trend by month for at-risk students.

Alternative datasets: retail orders, clinic appointments, or HR training.

Target Architecture

Python Agents (SQLite + Neo4j)

LLM_ONLINE_MODE=true means Python agents require LM Studio for planning/tool choice and final synthesis; connection failures are reported as errors. LLM_ONLINE_MODE=false skips LM Studio and returns an evidence-only answer from SQLite plus Neo4j. student_rag.agents.lmstudio is the Python API Tool Loop: LM Studio chooses tools each turn; Python executes SQL against SQLite and planned read-only Cypher against Neo4j. The Neo4j graph is built offline from Markdown docs using AutoSchemaKG.

AutoSchemaKG + Neo4j Knowledge Graph

Neo4j is preferred for the advanced showcase because AutoSchemaKG extracts entities, events, and relationships that are naturally graph-shaped.

MCP (SQLite + Neo4j Database Tools — Cursor / LM Studio Chat)

MCP remains database-facing: SQLite for operational records and read-only Neo4j graph context from AutoSchemaKG.

Scope Boundary

Use SQLite for operational records and Neo4j for graph context. Chart output stays a simple Vega-Lite JSON spec.

Runtime Mode

LLM_ONLINE_MODE controls Python-agent behavior. In online mode (true), LM Studio is required and connection failures are surfaced instead of hidden by fallback answers. In evidence-only mode (false), the agents skip LM Studio and synthesize responses directly from SQLite and Neo4j evidence.

Current MCP Tools

MCP is database-facing. It currently reads SQLite (student_management.sqlite) and should also read Neo4j through read-only graph tools for the AutoSchemaKG showcase. The same tool surface is used by Cursor and LM Studio Chat (local stdio or remote HTTP/SSE).

  • ask_student_management: routes common questions to structured SQLite queries.
  • get_schema_summary: returns tables, views, columns, and value hints.
  • run_sql: executes one read-only SQL query and returns warnings for suspicious values such as scholarship_candidate = 'yes'.
  • get_at_risk_students: returns high-risk and medium-risk students using the correct risk rules.
  • analyze_at_risk_students: returns at-risk rows plus metric interpretation guidance.
  • get_scholarship_candidates: returns scholarship candidates using scholarship_candidate = 1.
  • analyze_scholarship_candidates: returns scholarship rows plus metric interpretation guidance.
  • get_student_graph_context (planned): returns Neo4j policy, intervention, and risk-factor paths for one student.
  • query_knowledge_graph (planned): runs read-only Cypher against Neo4j.
  • generate_artifact: creates a Markdown table or Vega-Lite chart spec from SQL results.

Neo4j Graph Model

  • Nodes: Student, Course, Policy, RiskFactor, Intervention, Advisor, Program, SupportCase.
  • Relationships: HAS_RISK_FACTOR, TRIGGERS_POLICY, RECOMMENDS_INTERVENTION, ENROLLED_IN, ADVISED_BY, MENTIONED_IN.
  • Evidence properties: source_doc, evidence_text, confidence, term.

Schema Value Hints

  • student_risk_summary.risk_level uses high, medium, and low.
  • student_risk_summary.scholarship_candidate is an integer flag: 1 means yes and 0 means no.
  • student_risk_summary.term uses values such as 2026-Spring; there is no current_term.
  • fee_summary.status uses paid, partial, and overdue.
